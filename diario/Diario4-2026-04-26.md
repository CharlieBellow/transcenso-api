## Organização de Pastas (Application vs Domain)

"Aprendi que a pasta domain deve ser protegida e conter apenas as definições essenciais (Entidades e Contratos). Os Use Cases (Casos de Uso) vivem na camada de Application, pois eles são os roteiros que descrevem como a aplicação deve se comportar para atender a um pedido do usuário, orquestrando as entidades e os repositórios sem poluir o coração do domínio."

## Padronização de Estilo

- Decidi adotar o camelCase para nomenclatura de arquivos, trazendo minha experiência do Frontend (React) para o Backend. Isso mantém a consistência entre as camadas e facilita a identificação de classes e arquivos. O foco agora é garantir que os Use Cases de Gênero e Sexualidade 'conversem' entre si através dos repositórios para evitar duplicidade de siglas no TransCenso.

## Consolidação de Regras Cruzadas

"Implementei a validação cruzada nos Use Cases de Gênero e Sexualidade usando o operador ||. Agora, o sistema garante a unicidade das siglas em ambas as categorias, impedindo conflitos de dados. Também padronizei todo o projeto para camelCase, alinhando o backend com minha experiência em React, o que melhorou minha velocidade de leitura do código."

## Use Case de Pessoa e Integridade:
"Iniciei o createPersonUseCase, o mais complexo até agora. Diferente dos cadastros simples, aqui o Use Case atua como um validador de relacionamentos. Ele precisa consultar os repositórios de Gênero e Sexualidade para garantir que a pessoa está sendo vinculada a opções existentes, além de garantir a regra de ouro de que o CPF deve ser único no sistema."

## Estratégia de DTOs e Composição:
"Entendi que a Entidade (Props) deve focar em como os dados são salvos (IDs e datas automáticas). Já o DTO é uma visão para o usuário e pode ser 'composto' no Use Case. Aprendi que posso buscar informações de tabelas diferentes (Gênero e Sexualidade) dentro do mesmo Use Case para entregar um objeto completo e amigável para o Frontend, sem expor campos desnecessários como updatedAt de tabelas auxiliares."

## Inputs e Integridade:
"No CreatePersonInput, utilizo genderId e sexualityId. Isso reflete a realidade do sistema: o usuário escolhe uma opção no frontend, e o backend recebe a referência (ID) dessa escolha. O Use Case então valida se esse vínculo é real antes de permitir o 'nascimento' da nova Entidade Person, garantindo que o banco de dados nunca tenha referências órfãs ou inválidas."

## Composição de DTOs no Use Case:
"Entendi que a Entidade não deve ser responsável por montar DTOs complexos que dependem de outras entidades. A Entidade armazena apenas referências (genderId). A responsabilidade de 'enriquecer' o dado (transformar ID em objeto completo para o Frontend) é do Use Case, pois ele é o único que tem acesso a todos os repositórios necessários para buscar essas informações e montar o quebra-cabeça final."

## O Fim do Conflito de IDs:
"Descobri que o erro de tipagem ocorria porque eu tentava fazer a Entidade (que só tem IDs) se transformar em um DTO (que tem objetos completos). A solução foi remover o toDTO da Entidade e realizar a Assemblagem (montagem) do DTO final dentro do Use Case. Isso porque o Use Case é o único lugar que tem todas as peças do quebra-cabeça: a Entidade recém-criada e os objetos de Gênero/Sexualidade vindos dos seus respectivos repositórios."

## Ajustes de Ambiente e Permissões:

"Renomeei o projeto para TransCenso, refletindo corretamente o propósito de mapear dados. Enfrentei o primeiro desafio com Docker: o erro de permissão no docker.sock. Aprendi que isso é uma questão de segurança do Linux e que posso resolver usando sudo ou adicionando meu usuário ao grupo docker. O comando docker compose (sem hífen) agora é o padrão para subir o ambiente de desenvolvimento local."


## Incompatibilidade de Versões no Docker:

"Descobri através dos logs que o Postgres é extremamente rígido com a versão dos dados no volume. Tentar rodar uma imagem v15 em dados gerados pela v16 causa um erro FATAL. Aprendi que o docker logs é a ferramenta definitiva para diagnosticar por que um container entra em loop de reinicialização. Atualizei o projeto para a versão 16 para manter a compatibilidade e usei o down -v para garantir que o volume fosse recriado corretamente."

Aprendi a importância do padrão Mapper para traduzir dados entre o Prisma (detalhe técnico de persistência) e o Domínio (regras de negócio). Essa separação garante que minhas entidades permaneçam puras e que a troca de tecnologias de banco de dados no futuro seja menos dolorosa."

## Desvendando o Mapper e Repositório:

"Entendi que o Mapper resolve o conflito de tipos entre o que o banco de dados entrega e o que o meu domínio exige. O Repositório de Infra é o único lugar do sistema que 'sabe' que o Prisma existe. Para o resto do software (Use Cases), o repositório é apenas uma caixa preta que salva e busca dados, mantendo a arquitetura limpa e protegida contra mudanças tecnológicas."

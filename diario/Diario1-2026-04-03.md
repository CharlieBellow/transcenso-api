# 📘 Guia de Referência: Arquitetura e Boas Práticas (DDD + NestJS)

## 1. As Camadas do Projeto (A Cebola) - (03/04/26)

No DDD, dividimos o código em camadas para que a "lógica de negócio" não dependa de "tecnologias".

    - Domain (O Coração): Onde ficam as Entities. Contém as regras de ouro (Ex: Um currículo não pode ser vazio). Não conhece o NestJS nem o Banco de Dados.

    - Application (O Maestro): Onde ficam os Use Cases. É o passo a passo de uma tarefa (Ex: Salvar Currículo). No NestJS, aqui é onde moram os @Injectable() (Services).

    - Infrastructure (As Ferramentas): Onde ficam os Repositories. É o código que realmente fala com o Banco de Dados (TypeORM, Prisma, etc).

    - Interface (A Casca): Onde ficam os Controllers. É a porta de entrada que recebe o clique do usuário ou o JSON da internet.

## 2. Dicionário de Termos

Termo O que é? (Analogia) O que faz no Código?

- EntityA / Regra de Saúde/Negócio / Valida se os dados de um objeto são  
   reais e consistentes.

Use Case / O Procedimento Médico / Orquestra a tarefa: pede validação à
Entity e manda o Repository salvar.

Repository / O Arquivista da Clínica / Conhece os comandos do Banco de Dados
para buscar ou guardar as fichas.

Controller / A Secretária/Balcão / Recebe a requisição (Request) e
entrega a resposta (Response).

## 3. Fluxo de uma Ação (Ex: Salvar Currículo)

- O Controller recebe o currículo do React.

- O Controller passa os dados para o Use Case (SubmitResumeUseCase).

- O Use Case cria uma Entity (Resume) para garantir que os dados são válidos.

- O Use Case chama o Repository e diz: "Guarde este currículo aprovado".

- O Repository salva no Banco de Dados.

## 4. Dicas para as suas 4 Horas Semanais

- Pense em Tarefas (Use Cases): Em vez de criar um "serviço gigante", crie arquivos para tarefas específicas. É mais fácil de debugar e de retomar o raciocínio.

- Documente o "Porquê": No seu diário, anote por que você criou uma regra na Entity. Isso ajuda a internalizar o conceito.

- Isole o NestJS: Tente escrever sua Entity e seu Use Case como classes TypeScript puras primeiro. Depois, use o NestJS apenas para "plugar" tudo na internet.

### Nota-dev: "Hoje entendi que o Repository é apenas o funcionário que guarda as coisas. O Use Case é quem manda ele trabalhar. Parei na implementação da validação da Entity de Paciente."
